让 secondary 更会被用起来：长工具前先照应人类

本 PR 聚焦 Jason 指定的两个 P0：live notification trigger 与 secondary schema 文案。它不改 runtime 协议、不放宽 guardrail，只改变 agent 在关键时刻看到的提示。

为什么改

前一轮调研显示：secondary 运行时是完整、受限且成功率高的；真正问题是 agent 在规划时不够容易想起它。尤其当 notification 已经显示有人类消息，而下一步要跑 bash/daemon/web_search 等长工具时，旧提示没有强烈要求把回复/读取挂到 primary tool 的 secondary 字段上。

这次改了什么

  • 把 live notification guidance 改成明确触发：有人类消息 + 下一步是长 primary tool ⇒ 使用该 tool 的 secondary 字段先 reply/read。
  • 把 secondary schema 描述改为正向示例优先：先说明何时用、怎么用，再保留短调用禁用、工具白名单、动作白名单、禁止嵌套等 guardrail。

主要文件

  • src/lingtai_kernel/meta_block.py：ACTIVE tool result / IDLE synthesized notification 的 canonical notification payload guidance。
  • src/lingtai_kernel/base_agent/__init__.py：IDLE synthesized notification 文本摘要里的同步提示。
  • src/lingtai_kernel/base_agent/messaging.py：internal email unread digest 的 producer-specific instructions，新增 long primary 前 reply/read secondary 示例。
  • src/lingtai_kernel/secondary_tools.py：嵌套 schema 描述改成 positive-first examples。
  • tests/test_secondary_schema.py、tests/test_meta_block.py、tests/test_notification_sync.py、tests/test_layers_email.py：锁定新提示中的关键触发词和 guardrail。

多模型草稿如何影响最终方案

  • Claude Code 草稿强调：notification guidance 应从 “prefer” 提升到更明确的 imperative，并保留安全前提。
  • DeepSeek 草稿强调：schema 不需要改 shape，只重排描述即可，避免 runtime 风险。
  • GLM 草稿强调：除了全局 notification guidance，email digest 的 producer-specific instructions 也应给出 reply/read 示例。

最终实现吸收了三者共识：kernel-only、无协议改动、正向示例优先、长工具触发更明确。

风险与边界

  • 这是提示/文案层改动，不保证所有模型 100% 使用 secondary；后续可追加 missed-opportunity telemetry/nudge。
  • 提示更强后可能增加少量 proactive ack，但 guardrail 仍要求不要用于 routine short calls。
  • 外部 Telegram/WeChat/Feishu addon 的 producer-specific notification wording 未在本 PR 中修改；全局 kernel notification guidance 已覆盖 synthesized/active payload，addon-specific 深改可作为后续 PR。
