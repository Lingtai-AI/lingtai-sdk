redesignedPR #273

Tool error 还是 tool result：给 payload 加恢复元信息，而不是造 hard-stop

本报告取代旧版 “repeated tool-error hard-stop continuation” 说明。Jason 指出旧设计 overengineer；新版把修复收敛到既有 tool-result 数据路径。

结论

删掉特殊控制流

移除 _REPEATED_IDENTICAL_ERROR_HARD_STOP_COUNT、_build_repeated_tool_error_summary、repeated_tool_error_hard_stop / continuation 分支。重复错误不再触发 session.send(str)。

保留普通 tool flow

工具执行结果仍经 session.send(tool_results) 回到 LLM。真正无限循环仍由通用 tool-loop / emergency limits 兜底。

增强错误 payload

在既有 PR #246 的 flat error fields 上，新增嵌套 tool_error block，集中给 summary、reason、retryability、arg_keys 与恢复建议。

为什么这样改

原问题不是“第三次同样错误必须被 kernel 打断”，而是错误结果对模型不够解释性：模型看到空泛 error 时容易原样重试。正确修复是在 tool error payload 本身说明失败原因与下一步，而不是在 turn 层新增 hard-stop / notification / text continuation。

模型可见 payload 形状

{
  "status": "error",
  "message": "chat_id must be integer",
  "error_type": "telegram",
  "error_phase": "tool_returned_error",
  "tool_name": "telegram",
  "tool_call_id": "tg-err",
  "tool_trace_id": "tg-err",
  "tool_args": {"action":"send","chat_id":"","text":""},
  "arg_keys": ["action","chat_id","text"],
  "retryable": "unknown",
  "_tool_error_payload_version": 1,
  "tool_error": {
    "version": 1,
    "summary": "chat_id must be integer",
    "reason": "telegram failed during tool_returned_error: chat_id must be integer",
    "retryable": "unknown",
    "guidance": [
      "Do not blindly retry the same tool call unchanged.",
      "Use the error message, phase, and argument keys to correct parameters or switch strategy.",
      "If the failure depends on mutable external state, read the current state before retrying.",
      "If the cause cannot be corrected, report the failure and ask for direction instead of looping."
    ]
  }
}
