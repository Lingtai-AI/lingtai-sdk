Bash manual — async rule for long-running agent CLIs

Ready for review Scope: docs only

Summary

Documents a normative rule in the bash manual: a long-running agent/coding CLI session (claude -p, codex exec, opencode run, the Cursor agent CLI, or any sub-agent that may think and run tools for minutes) must never be run as a synchronous bash call. Synchronous bash is reserved for short, deterministic commands; long-running child CLIs use bash(async=true) and poll the returned job_id.

Problem

An audit of LingTai trajectories found agents blocking themselves by running long-lived child agent CLIs (e.g. claude -p) as synchronous bash calls. A synchronous call holds the turn until the child exits — which can be minutes — so the parent stays ACTIVE and stops seeing channel notifications (mail, refresh, interrupts) for the whole duration. The bash tool already supports async=true + action="poll"; the manual just never told agents to use it for this case. The old text said ordinary one-off commands can use bash directly and only routed scheduled work, leaving the long-running-CLI case unaddressed.

What changed

One file edited: src/lingtai/core/bash/manual/SKILL.md (top-level router; version bumped 1.2.0 → 1.3.0). Changes are additive — no existing guidance removed.

  • Router intro now distinguishes short, deterministic synchronous commands from long-running agent CLIs.
  • New router-table row pointing the long-running-CLI case at the resident rule (bash(async=true) + poll).
  • Quick decision tree gains an explicit step: long-running agent/coding CLI ⇒ never synchronous, use async + poll. Subsequent steps renumbered.
  • New first entry under "Core rules to keep resident": the normative rule plus a text example showing start (async=true → job_id), poll, and cancel, and why this keeps the agent responsive and avoids ACTIVE blockage.

Non-goals

  • No code changes — the async/poll/cancel bash surface already exists (BashManager in src/lingtai/core/bash/__init__.py).
  • No new nested reference file; the rule belongs in the resident top-level router, kept short.
  • No changes to scheduled-work / notification / debugging references.
  • No mutation of historical reports.
