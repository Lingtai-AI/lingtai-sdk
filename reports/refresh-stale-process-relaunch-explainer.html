Refresh stale-process relaunch fix

Conclusion: refresh watcher relaunch can permanently fail when a stale same-agent lingtai run <agent-dir> process remains visible after the old process stops making progress. This patch teaches the watcher to terminate only a confirmed stale same-workdir duplicate, then retry relaunch.

Observed production failure

During dev-1 recovery on 2026-06-04, mimo-1 had stale status/history, no heartbeat file, and refresh telemetry ended with twelve relaunch attempts failing because the CLI duplicate-process guard found an old process in the same workdir.

error: another lingtai agent is already running in /Users/huangzesen/work/projects/lingtai-dev/dev-1/.lingtai/mimo-1
  PID 72272: ... Python -m lingtai run /Users/huangzesen/work/projects/lingtai-dev/dev-1/.lingtai/mimo-1
  If this is a stale process, kill it first.
...
{"type": "refresh_failed_permanent", "attempts": 12}

Patch

  • Adds duplicate-guard stderr parsing inside the refresh watcher relaunch script.
  • Verifies the blocker is still a same-workdir lingtai run process before touching it.
  • Checks heartbeat freshness; a fresh heartbeat logs refresh_watcher_duplicate_alive and is not killed.
  • If no fresh heartbeat exists, sends SIGTERM, waits briefly, escalates to SIGKILL if needed, logs refresh_watcher_stale_duplicate_*, then retries the normal relaunch loop.
  • Updates base_agent/ANATOMY.md with the new refresh stale-process recovery contract.

Safety properties

RiskMitigation
Killing a healthy agentCleanup only runs after the CLI duplicate-process guard rejects relaunch and only for a process whose command line contains lingtai run + wd. A fresh heartbeat (<60s) blocks cleanup.
Killing unrelated processPID is extracted from the guard output, then revalidated with ps -p PID -o command= before any signal.
Silent actionEvery cleanup path logs structured events: refresh_watcher_stale_duplicate_terminate, gone, killed, or error variants.
