PR explainer

MiMo Code and Qwen Code as daemon backends

This change expands LingTai daemon delegation so agents can send isolated work to MiMo Code or Qwen Code CLIs, using the same observable daemon run directories, notifications, and validation paths as existing CLI backends.

New backend
mimocode / mimo

OpenCode-family JSON runner using mimo run --format json.

New backend
qwen-code / qwen

Headless Qwen runner using qwen --yolo -p.

What changed

AreaChange
Backend schemaAdded mimocode, alias mimo, qwen-code, and alias qwen to the daemon backend enum and help text.
Alias handlingIntroduced backend canonicalization so short names persist as canonical mimocode / qwen-code daemon entries.
MiMo CodeReused the defensive OpenCode-family JSON parser with executable mimo, session storage key mimocode_session_id, and resume via mimo run --session ... --format json.
Qwen CodeAdded a headless runner that captures stdout/stderr from qwen --yolo -p <prompt> and marks daemon completion from captured output.
backend_optionsExtended validation so OpenCode-family backends reserve --format, and Qwen reserves prompt/approval flags owned by LingTai's harness.
DocsUpdated daemon anatomy, CLI backend manual, and i18n help strings across English, Chinese, and Wenyan locales.

Review notes

  • The local machine used for development did not have mimo or qwen installed, so tests mock subprocess boundaries and assert command construction, validation, and state transitions.
  • qwen-code intentionally owns --yolo and -p so daemon runs are non-interactive and prompt placement remains deterministic.
  • The OpenCode runner was parameterized rather than duplicated, keeping MiMo Code behavior aligned with future OpenCode-family parser fixes.
