Daemon 并发上限与 init.json 覆盖修复

目标：默认 daemon 并发上限从 10 提到 100，并确保 manifest.capabilities.daemon.max_emanations 能通过 init.json 覆盖默认值。

结论

已修复两个问题。 原因不是单纯默认值低，而是 active preset materialization 会把 agent 自己写在 init.json 里的 core capability kwargs 覆盖掉；因此即使写了 daemon.max_emanations=30/50，refresh 后仍回到默认。

默认值
Daemon 默认 max_emanations：10 → 100
可配置
init.json 的 daemon override 在 preset refresh 后会保留
兼容性
非 core 的可选 capability 仍由 preset 决定，不被 init-only 项污染

Root cause

  1. DaemonManager 和 daemon.setup() 的默认参数写死为 10。
  2. 更隐蔽的问题：materialize_active_preset() 会把 manifest.capabilities 换成 active preset 的 capabilities。Codex preset 不显式列 daemon，但 daemon 是 core default；后续 apply_core_defaults() 重新加回 daemon={}，于是 init.json 的 max_emanations 丢失。

修复设计

场景新行为
Preset 也启用了同名 capability按 key 合并，preset 提供默认，init.json kwargs 优先。
Preset 没列出 core default capability（如 daemon）保留 init.json 对 core default capability 的 kwargs，因为这些 capability 本来就会启动。
Preset 没列出非 core 可选 capability（如 vision/web_search）仍然丢弃 init-only 可选 capability，保持“切换 preset = 切换能力集合”的语义。
skills.paths保留原有特殊规则：preset paths 在前，init.json paths append + 去重。

主要改动文件

  • src/lingtai/core/daemon/__init__.py：默认 max_emanations 改为 100。
  • src/lingtai_kernel/presets.py：preset materialization 支持 core default capability kwargs carry-forward。
  • src/lingtai/agent.py / src/lingtai/cli.py：调用 materialization 时注入 CORE_DEFAULTS，避免 kernel 包反向依赖 wrapper。
  • tests/test_preset_materialization.py、tests/test_daemon.py、tests/test_presets.py：新增/更新回归测试。
  • src/lingtai/ANATOMY.md、CLAUDE.md：同步说明。
