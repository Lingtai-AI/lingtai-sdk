Soul Flow: routed past-self selection

This PR turns soul-flow past voices from fixed/random fanout into a sparse, relevance-routed consultation batch with bounded serendipity and smaller past-self context budgets.

LingTai kernel intrinsics/soul MoE-inspired routing token budget

Conclusion first

Before
Soul flow ran the current-self insight plus up to K randomly sampled past snapshots.
After
K becomes a maximum. Past snapshots are selected by lexical relevance to the current diary, with at most one probabilistic serendipity voice.
Safety / responsiveness
Pending non-soul notifications suppress past voices so human-facing work is not delayed by expensive introspection.
Budget
Current-self insights retain the 70% window; past-self snapshots use 30% to reduce context burn for selected memories.

Routing model

current diary cue
  ├─ shared current-self: insights (70% context budget)
  └─ past-self pool
       ├─ guard: no past voices if non-soul notification is pending
       ├─ guard: no past voices if diary is empty
       ├─ routed: top overlap between diary tokens and snapshot molt_summary
       ├─ serendipity: at most one random extra if budget remains, 25% chance
       └─ fallback: random sample if diary exists but no lexical route matches

Why this shape

The design borrows the engineering intuition of sparse activation: many memory experts can exist, but each fire should activate only the few most useful ones. The current self remains the shared path; past selves become routed experts, not a fixed random fanout.

It also follows a practical responsiveness rule: when a non-soul notification exists, a human or external channel may be waiting, so the system should avoid spending extra calls on historical introspection.

Known boundaries

  • Routing is intentionally lightweight lexical overlap over top-level molt_summary; it does not load full snapshots before selection.
  • Cache-friendly prompt layout is not fully redesigned here; this PR only takes the first budget step by reducing past-self context to 30%.
  • The non-soul notification guard is conservative and may suppress past voices for machine-generated channels too; route logs should make this observable.
